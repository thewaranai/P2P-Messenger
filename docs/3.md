# Функциональные и нефункциональные требования

## Функциональные требования

Система должна обеспечивать выполнение следующих функций:

| ID | Функция | Описание |
|----|---------|----------|
| FR-01 | Регистрация пользователя | Пользователь создаёт учётную запись с уникальным именем и паролем. |
| FR-02 | Вход в систему | Аутентификация по имени и паролю. |
| FR-03 | Отображение списка контактов | После входа отображается список ранее добавленных контактов. |
| FR-04 | Поиск доступных пользователей | Сканирование локальной сети для обнаружения других активных пользователей. |
| FR-05 | Подключение к выбранному пользователю | Установка P2P-соединения с выбранным контактом. |
| FR-06 | Отправка текстовых сообщений | Передача текста через активное соединение. |
| FR-07 | Получение сообщений в реальном времени | Приложение прослушивает сокет и отображает входящие сообщения. |
| FR-08 | Отображение времени отправки | Каждое сообщение сопровождается временной меткой. |
| FR-09 | Хранение истории переписки | Все сообщения сохраняются локально в БД. |
| FR-10 | Уведомление о новом сообщении | Всплывающее уведомление при получении нового сообщения. |
| FR-11 | Отображение статуса соединения | Индикатор подключения (онлайн/офлайн). |
| FR-12 | Завершение соединения | Пользователь может разорвать соединение в любой момент. |

## Нефункциональные требования

### Производительность

- Время доставки сообщения не должно превышать **1 секунду** при стабильном соединении.
- Приложение должно запускаться не дольше **3 секунд**.

### Надёжность

- Приложение должно корректно обрабатывать разрывы соединения и автоматически уведомлять пользователя.
- В случае ошибки соединения должна быть предусмотрена возможность повторного подключения.
- Приложение не должно терять сохранённые данные при нештатном завершении.

### Удобство использования (Usability)

- Интерфейс должен быть интуитивно понятным и соответствовать Material Design.
- Все действия (отправка, подключение) должны выполняться не более чем за 2 шага.

### Поддерживаемость

- Код должен быть структурирован (MVP/MVVM) для лёгкого расширения.
- Архитектура должна позволять добавление новых типов сообщений (файлы, голос) в будущем.

### Совместимость

- Приложение должно работать на устройствах с **Android 8.0 (API 26) и выше**.
- Поддержка экранов различных размеров и ориентаций.

### Безопасность

- Пароли хранятся в хэшированном виде.
- Передача данных по P2P-каналу без промежуточных серверов снижает риск перехвата.

### Технические ограничения

- Приложение использует локальную сеть (Wi-Fi) для обнаружения и соединения.
- Используется язык **Kotlin**, фреймворк **Android SDK**, СУБД **SQLite**.
